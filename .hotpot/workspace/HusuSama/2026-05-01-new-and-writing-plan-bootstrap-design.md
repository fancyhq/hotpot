# `/new` 命令与 `writing-plan` skill — 设计 Spec

> Generated 2026-05-01 by HusuSama  
> Topic: bootstrap of hotpot's "new task" workflow

## 1. 目标

为 hotpot 项目搭建"新任务工作流"的两个基础组件：

- 一个手动触发的 `/new` slash command，引导用户走 brainstorming 流程，期间不写文件。
- 一个 `writing-plan` skill，仅由 `/new` 在用户批准设计后调用，把"设计 + 实施计划"合并写入一份 markdown 到当前开发者的 workspace 目录。

这两个组件是 hotpot CLI 后续打包进二进制并通过 `hotpot init -a <agent>` 分发的 **公共模板源** 之一，本次仅完成 Claude Code 一条目标路径的可用副本。

## 2. 范围

**In scope**

- 创建以下 4 个文件，源文件用英文撰写：
  - `commands/new.md`（模板源）
  - `skills/writing-plan.md`（模板源，平铺，无子文件夹）
  - `.claude/commands/new.md`（Claude Code 工作副本，与模板源逐字相同）
  - `.claude/skills/writing-plan.md`（同上）
- 让 `/new` 在 Claude Code 中可被手动触发，引导式 brainstorming。
- 让 `writing-plan` skill 在 `/new` 流程末尾被调用，把产物写入 `.hotpot/workspace/<git user.name>/YYYY-MM-DD-<topic>.md`。

**Out of scope**

- hotpot CLI 本体（`hotpot init` 的实现、二进制打包、跨 agent 路径替换逻辑）。
- 在 frontmatter 里使用 `allowed-tools` 这类 Claude 特有字段（保留跨 agent 兼容）。
- 修改 `.gitignore`（workspace 目录纳入 git 跟踪，团队共享 brainstorming 产出）。
- 模板源到 `.claude/` 副本的自动同步脚本（dev 期手工同步即可）。

## 3. 架构

### 3.1 文件布局

```
hotpot/
├── commands/
│   └── new.md                   # 模板源：/new 命令
├── skills/
│   └── writing-plan.md          # 模板源：writing-plan 技能
└── .claude/
    ├── commands/
    │   └── new.md               # Claude Code 工作副本
    └── skills/
        └── writing-plan.md      # Claude Code 工作副本
```

### 3.2 调用链

```
User types `/new` (或 `/new <短描述>`)
   │
   ▼
Claude Code 加载 .claude/commands/new.md，按其指示进入 brainstorming 流程
   │  （HARD-GATE：本阶段只对话，不写文件、不实现、不调写工具）
   ▼
AI 与 User 一问一答，呈现 2-3 方案，分节呈现设计
   │
   ▼
User 批准设计
   │
   ▼
AI 按 new.md 末尾指示，读取并执行 .claude/skills/writing-plan.md
   │
   ▼
writing-plan skill 取 git user.name → 规范化 topic → 检查重名 → 写入文件
   │
   ▼
落地：.hotpot/workspace/<username>/YYYY-MM-DD-<topic>.md
   │
   ▼
AI 告知用户文件路径，工作流结束（实施阶段由用户后续手动决定）
```

## 4. `/new` 命令规格

**触发**：用户在 Claude Code 中输入 `/new` 或 `/new <一句话任务描述>`。

**模板源语言**：英文。

**命令文档要让 AI 做的事（按文档指示，不靠 frontmatter 工具白名单）**：

1. **声明阶段**：用一句话告诉用户当前进入 "new task brainstorming"，并说明 HARD-GATE：本阶段只对话，不写文件、不实现、不 scaffold。
2. **了解项目上下文**：可以使用只读工具（`Read / Glob / Grep`）阅读仓库文件、最近 commit、相关 docs。**禁止使用** `Write / Edit / NotebookEdit` 或任何会修改磁盘的 `Bash` 命令。
3. **任务澄清**：一次问一个问题（multiple-choice 优先），了解目的 / 约束 / 成功标准。
4. **方案探索**：提出 2–3 种方案 + tradeoff + 推荐。
5. **分节呈现设计**：按 architecture / components / data flow / error handling / testing 等维度，每节得到 user 认可后再进下一节。
6. **得到用户明确批准**后，**调用 writing-plan skill** —— 通过读取并执行 `<agent skills dir>/writing-plan.md`（Claude Code 上即 `.claude/skills/writing-plan.md`）。
7. **不依赖 superpowers**：流程在文档自身写清楚，不引用 `superpowers:brainstorming` 或其他外部 skill 名。

**对话语言**：跟随用户。用户用中文就用中文回。

**前置参数**：用户可以在 `/new` 后追加一句话主题（用作 topic 的初值），不强制。

**终态**：调用 writing-plan skill。**不**调用 frontend-design / mcp-builder / 其他实现 skill。

**Frontmatter（最小集，跨 agent 通用）**：

```yaml
---
description: Start a new task — guided brainstorming, then save the spec+plan to .hotpot/workspace/<your-name>/.
---
```

不写 `allowed-tools`（用文档内约束代替）。

## 5. `writing-plan` skill 规格

**触发**：仅由 `/new` 在用户批准设计后调用。**不**作为 auto-trigger skill 自动加载（即使 Claude Code 看见它，也只在 `/new` 链路结尾被引用执行）。

**模板源语言**：英文。

**产物**：单个 markdown 文件（spec + 实施计划合并），保存到：

```
.hotpot/workspace/<username>/YYYY-MM-DD-<topic>.md
```

- `<username>` = `git config user.name`（local 优先，回退到 global）
- `YYYY-MM-DD` = 今天日期（生成时取）
- `<topic>` = 用户给的简短主题，做 kebab-case 规范化（小写 / 空格转 `-` / 去除 `[a-z0-9-]` 之外的字符）

**输出文档语言**：跟随用户当下对话语言（不是英文模板，是给开发者看的产物）。

**输出文档内部结构**：

```markdown
# <Title>

> Generated by hotpot writing-plan skill on YYYY-MM-DD by <username>

## 1. Design (Spec)
- Goal
- Constraints
- Success Criteria
- Architecture & Components
- Data flow / Error handling / Testing strategy

## 2. File Structure
- 列出本计划将创建 / 修改的文件，每个文件一句话职责说明

## 3. Implementation Steps (Plan)
- 按 superpowers writing-plans 风格的 bite-sized task：
- 每个 Task: Files 段（Create / Modify / Test）+ 一系列 - [ ] Step
- 每步 2–5 分钟可完成
- 每步要有可粘贴的代码 / 命令；禁止 "TBD" / "类似上一步" / "implement later"

## 4. Acceptance
- 怎么算这个 plan 实施完成了（test commands、manual checks）
```

**写入流程（写在 skill 文档中，让 AI 按这个执行）**：

1. **取 username**：`git config user.name`。如果空 / 未配置 → 报错并提示用户先 `git config --global user.name "<your-name>"`，本次不写文件。
2. **取日期**：`date +%Y-%m-%d`，回退到对话上下文中的当前日期。
3. **规范化 topic**：从 brainstorming 阶段确定的主题取，做 kebab-case。
4. **目录确保存在**：`mkdir -p .hotpot/workspace/<username>/`。
5. **冲突检测**：如果 `YYYY-MM-DD-<topic>.md` 已存在 → 提示用户，让其选择 ① 覆盖 ② 改 topic ③ 加 `-2`/`-3` 后缀（自动找最小可用序号）。**不静默覆盖**。
6. **Write 整个文档**。
7. **完成提示**：告知用户文件路径，问"是否要进入实施阶段"——这一步只是提示，不强行 invoke 其他 skill；实施由用户手动决定。

**Self-review（写在 skill 文档中，要求 AI 在写完文档后自检）**：

- 占位符扫描：`TBD` / `TODO` / "fill in later" / "类似上面"——必须修掉。
- 内部一致性：早期 task 引入的方法名 / 类型，后续 task 引用时必须保持一致。
- spec 覆盖：File Structure 表中每个文件，是否在某 task 里被实际使用？
- scope check：是不是太大需要拆成多个 plan？如果是，告诉用户。

**Frontmatter（最小集）**：

```yaml
---
name: writing-plan
description: Write an approved design + implementation plan into .hotpot/workspace/<your-git-name>/. Only invoked from the /new command after the user approves the design.
---
```

## 6. 跨 agent / 模板策略

**当前阶段（dev 期）的同步**：

- 模板源 = single source of truth。`.claude/` 副本逐字相同，不做任何 agent 特定改写。
- 任何修改都先改模板源，再手工同步到 `.claude/`（dev 同步脚本不在本次 scope）。

**模板内部对路径的引用**：

- `/new` 命令文档结尾说："Read and follow the instructions in the writing-plan skill (located at the agent's skills directory; for Claude Code that's `.claude/skills/writing-plan.md`)。" —— **不写绝对路径硬编码**，提示性描述足以让 AI 找到。
- `writing-plan` skill 文档说"save to `.hotpot/workspace/<username>/`"——这是 hotpot 自己定义的固定路径，跨 agent 通用，无需替换。

**Frontmatter 最小化**：

- 仅保留所有 agent 都识别的 `name` / `description` 字段。
- 不写 `allowed-tools`（Claude 特有），不写 `model`、`disable-model-invocation` 等。

**未来 hotpot init 意图（仅记录，不实现）**：

- `hotpot init -a claude`：把 `commands/*.md` → `.claude/commands/`，`skills/*.md` → `.claude/skills/`。
- `hotpot init -a opencode`：复制到 OpenCode 对应目录（路径将来确定）。
- `hotpot init -a opencode,claude`：多目标。
- 路径文案中"for Claude Code that's …"这类提示，hotpot 可在拷贝时按 agent 改写——但**本次只确保 Claude 这条路顺**。

## 7. 接受标准 / Acceptance Criteria

- [ ] 4 个文件存在并内容一致（commands/skills 与 .claude/ 副本逐字相同）。
- [ ] 模板源文件为英文。
- [ ] 用户在项目中输入 `/new` 时，命令被 Claude Code 识别并启动 brainstorming 对话。
- [ ] brainstorming 阶段 AI **不**写文件、不调写工具——靠文档 HARD-GATE 约束。
- [ ] 用户批准设计后，AI 通过执行 `.claude/skills/writing-plan.md` 调用 writing-plan skill。
- [ ] writing-plan skill 取 `git config user.name`，落地到 `.hotpot/workspace/<username>/YYYY-MM-DD-<topic>.md`。
- [ ] 重名时不静默覆盖，给用户三选一。
- [ ] 产出文档结构包含 Design / File Structure / Implementation Steps / Acceptance 四节。
- [ ] 产出文档语言跟随用户当下对话语言。
- [ ] `git config user.name` 为空时，skill 报错并阻止写入。
