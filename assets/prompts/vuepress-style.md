## VuePress Task File Writing Conventions (Extensions Catalog)

This file is loaded by `/hotpot:new` whenever VuePress is enabled. It
lists the **extensions the VuePress default theme supports on top of
standard GFM markdown** so you know what is available when you write a
task file.

**This is NOT a restrictive whitelist.** Standard markdown syntax
(headings, paragraphs, bold `**...**`, italics `*...*`, inline code,
blockquotes `>`, ordered / unordered lists, task lists `- [ ] todo` /
`- [x] done`, horizontal rules `---`, tables, images, links, inline
HTML, etc.) all work normally and need no special handling. The
sections below only describe "what *extra* you can use" plus "what to
avoid"; anything not mentioned defaults to safe standard markdown.

### 1. Frontmatter (YAML block at top of file)

Use any subset of these fields. Only `title` is required; the rest are
optional. `category` / `tag` help the default theme group entries on
index pages.

````markdown
---
title: Task title (one line)
description: One-line description of what the task aims to do
date: 2026-05-17
category: [Task]
tag: [<relevant-keywords>]
---
````

Do NOT write `sidebar`, `prev`, `next`, or `permalink` — those are
typically auto-generated by VuePress sidebar logic and manual values
conflict with the auto-config.

### 2. Custom containers

For tips, warnings, and callouts, prefer VuePress container syntax over
emoji + blockquote — the default theme styles them consistently:

````markdown
::: tip Tip
...
:::

::: warning Caution
...
:::

::: danger Danger
...
:::

::: info Info
...
:::

::: details Click to expand
...
:::
````

Full markdown is supported inside containers.

### 3. Code blocks

Line numbers + line highlighting (native to the default theme):

`````markdown
```rust {2,4-6}
fn main() {
    println!("highlighted");  // this line is highlighted
    let x = 1;
    let y = 2;                // these three lines
    let z = 3;                // are all highlighted
    println!("{}", x + y + z);
}
```
`````

Optional: code block title (rendered as a tab by the default theme):

`````markdown
```rust title="src/main.rs"
...
```
`````

### 4. Links

- In-project links use relative paths + `.md` suffix:
  `[spec](./vuepress-style.md)` — VuePress rewrites it to `.html`
  automatically.
- External links are normal: `[label](https://...)`.
- Do NOT use `[[wiki-style]]` internal links — the default theme does
  not parse them.

### 5. Tables

Standard GFM tables work, including alignment markers:

````markdown
| Field | Type   | Description |
| ----- | ------ | ----------- |
| id    | string | Task ID     |
| date  | string | YYYY-MM-DD  |
````

### 6. Disabled list

The following features must NOT be used (either the default theme does
not render them, or they cannot be reliably emitted by an AI):

- Vue components: `<Badge>` / `<RouteLink>` / `<CodeGroup>` /
  `<CodeTabs>` / `<Tabs>` etc. They depend on theme/plugin runtime and
  fall back to raw tags in plain-markdown rendering.
- `[[toc]]` inline table-of-contents — default-theme support is
  inconsistent.
- `<<< @/path/to/file` code import — requires extra
  markdown-power plugin config not enabled in this template.
- Mermaid diagrams (` ```mermaid ` blocks) / math expressions
  (`$...$` etc.) — depend on plugins not enabled in this template.
- Color / style raw HTML such as `<span style="color:red">` — same
  reason.

### 7. Hotpot mandatory section anchors

Hotpot still requires the task `.md` to contain `## Task` / `## Plan` /
`## Execution Instructions` and the other structural sections.
**These English anchors MUST NOT be renamed or translated** (not even
when the project `output-language` is set to a non-English value).
VuePress will render them as ordinary `<h2>` headings — no special
handling needed — but their presence is the hard contract
`/hotpot:execute` relies on to parse the task file.

### General principles

- Standard GFM markdown (including task lists `- [ ]`) is **safe by
  default**. Write the way you would write plain markdown.
- The extensions in sections 1–5 are "use them if they help" — they are
  **not mandatory**. Simple tasks are fine with pure markdown.
- The disabled list in section 6 is "using them will break things" —
  those genuinely must be avoided.
- When unsure whether a non-standard extension is supported by VuePress,
  fall back to the most basic markdown.
- The primary goal of the task file is correct parsing by
  `/hotpot:execute` (structural English anchors intact); a pretty render
  in VuePress is secondary.
